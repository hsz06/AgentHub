# 01. 产品定位与目标

## 1. 背景

传统 AI 编程工具通常以 “单个 Agent + 单次任务” 为主，用户需要在不同工具、终端、编辑器和网页之间切换。AgentHub 的设定是把这些能力组织到一个 IM 工作空间中：用户通过会话表达需求，通过 @ 提及选择 Agent，通过审批控制文件变更，通过 Artifact 查看最终产物。

本课题关注的不是做一个普通 Chatbot，而是验证 “多 Agent 是否能像团队成员一样在同一个上下文中协作”。

## 2. 产品定位

AgentHub 定位为本机可运行的多 Agent 协作 Demo：

- **主入口是 IM**：会话、消息、参与者、引用、@ 提及和消息流是主要交互。
- **工作对象是 workspace**：Agent 的代码修改必须围绕绑定 workspace 发生。
- **产物必须可审查**：代码 Diff、文档、网页预览、部署状态都要回到聊天流或控制中心。
- **执行必须有边界**：CLI Agent 在临时副本中运行，正式 workspace 只接受用户批准后的变更。

## 3. 目标用户

|用户类型|典型目标|关键需求|
|---|---|---|
|学生 / 答辩者|展示 AI 协作开发平台的完整链路|需求、架构、Demo、协作记录可讲清楚|
|前端开发者|快速生成页面并调整视觉效果|聊天式需求输入、代码预览、Diff 审批、Artifact 预览|
|全栈开发者|让多个 Agent 分工处理复杂任务|群聊调度、Workspace 绑定、命令审批、本机部署|
|产品 / 评审者|查看过程和结果，不直接写代码|聊天记录、审批队列、产物预览、部署状态|

## 4. 核心价值

### 4.1 把 Agent 变成联系人

用户不需要记住各种 CLI 参数或 API 调用方式，只需要在聊天中选择 Agent：

- Claude Code CLI：适合本机代码修改。
- Codex CLI：适合代码生成和评审。
- OpenCode CLI：适合本地开发任务。
- 自建 Agent：适合产品、测试、文档、前端、后端等角色。

### 4.2 把复杂任务变成群聊协作

在群聊中，Orchestrator 可以根据用户意图拆分任务：

```text
用户：做一个任务管理页面，要求好看、可预览、补文档。
Orchestrator：
1. 前端 Agent 负责页面和交互。
2. 测试 Agent 负责验收点。
3. 文档 Agent 负责 README 和使用说明。
4. CLI Agent 在 workspace 副本中实际修改文件。
```

### 4.3 把文件变更变成审批

Agent 生成代码不等于直接写入正式项目。AgentHub 的产品边界是：

```text
Agent 修改临时 workspace
  → 平台生成 Diff
  → 用户预览
  → 用户批准
  → 写回正式 workspace
```

这样可以让 AI 生成能力和人工控制权同时存在。

### 4.4 把产物放回聊天上下文

Agent 的产出不只是纯文本，还包括：

- Diff 卡片
- Web 预览
- Markdown 文档
- Slides 预览
- 附件下载
- 部署状态
- 命令运行日志

这些产物必须和原始需求在同一个会话中可追溯。

## 5. 产品非目标

|非目标|原因|
|---|---|
|公网多租户平台|本课题以本机可运行 Demo 为主，生产安全不是 P0|
|完整云端容器平台|本机 Runner 已能验证 Agent 协作主链路|
|移动端完整代码编辑|移动端定位为轻量审批和预览，不承担复杂 IDE 能力|
|直接复制 CLI 二进制|Claude Code / Codex / OpenCode 应使用用户本机安装版本|
|绕过审批自动写盘|这会破坏 AgentHub 的安全边界和可解释性|

## 6. 成功标准

产品成功不以页面数量衡量，而以端到端链路衡量：

1. 用户可以创建单聊和群聊。
2. 用户可以 @ 已有 Agent 或配置自建 Agent。
3. Agent 可以基于上下文回复。
4. CLI Agent 可以围绕 workspace 工作。
5. 文件变更可以生成审批，审批前正式文件不变。
6. 产物可以在聊天流或控制中心预览。
7. 技术文档能解释核心架构和边界。
8. AI 协作记录能证明开发过程有 Spec、rules、skill 沉淀。
